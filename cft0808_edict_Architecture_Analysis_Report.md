## 一、项目概述

### 1.1 核心定位

edict（三省六部）用**中国古代唐朝的三省六部制度**重新设计 AI 多 Agent 协作架构：

> "我用 1300 年前的帝国制度，重新设计了 AI 多 Agent 协作架构。结果发现，古人比现代 AI 框架更懂分权制衡。"

核心创新：**制度性审核**（门下省专职审核，可封驳）+ **实时看板**（军机处）+ **完全可干预**。

### 1.2 解决的问题

现有 Multi-Agent 框架的通病：
- **没有审核**：做完就交，没有 QA 部门，工程师写完代码直接上线
- **没有可观测性**：不知道任务流转到哪一步
- **不能干预**：任务跑起来就不能停

### 1.3 三省六部的解法

```
皇上 (你) → 太子 (分拣) → 中书省 (规划) → 门下省 (审核) → 尚书省 (派发) → 六部 (执行) → 回奏
```

不是"你们几个 AI 自己聊"，而是**严格的分权制衡**：

| 角色 | 职责 |
|------|------|
| **太子** | 消息分拣——闲聊自动回复，旨意才建任务 |
| **中书省** | 规划——子任务拆解 |
| **门下省** | 审核——**可封驳**，不合格的打回重做 |
| **尚书省** | 派发——任务分配给六部 |
| **六部** | 执行——吏/户/礼/兵/刑/工 |
| **早朝官** | 兼容角色 |

### 1.4 关键指标

| 指标 | 数值 |

| 技术栈 | Python + React + stdlib only backend |

### 2.2 核心差异化：门下省审核

这是 edict 的杀手级功能：

```
中书省规划方案 → 门下省审查 → 
  ├─ 合格 → 尚书省派发 → 六部执行
  └─ 不合格 → 封驳 → 打回中书省重做 → 重新审查（循环）
```

**为什么这个设计重要**：

> CrewAI 和 AutoGen 的 Agent 协作模式是"做完就交"——没有人检查产出质量。就像一个公司没有 QA 部门，工程师写完代码直接上线。

**门下省专职干这件事**：
- 📋 审查方案质量——中书省的规划是否完备？子任务拆解是否合理？
- 🚫 封驳不合格的产出——不是 warning，是直接打回重做
- 🔄 强制返工循环——直到方案达标才放行

这不是可选的插件，**它是架构的一部分**。

## 三、部署与架构

### 3.1 安装方式

| 方式 | 命令 |
|------|------|
| **Docker 一键** | `docker run -p 7891:7891 cft0808/sansheng-demo` |
| **完整安装** | `git clone && ./install.sh && ./start.sh` |

### 3.2 架构特点

- **后端零外部依赖**：stdlib only，不依赖 LangChain 等框架
- **前端**：React 18
- **依赖**：OpenClaw
- **部署**：systemd 支持

### 3.3 安装脚本做的事

`install.sh` 自动完成：
1. 创建全量 Agent Workspace（含太子/吏部/早朝）
2. 写入各省部 SOUL.md（角色人格 + 工作流规则 + 数据清洗规范）
3. 注册 Agent 及权限矩阵到 `openclaw.json`
4. 符号链接统一数据
5. 设置 Agent 间通信可见性
6. 同步 API Key 到所有 Agent
7. 构建 React 前端
8. 初始化数据目录 + 首次数据同步
9. 重启 Gateway 使配置生效

## 五、洞察总结

### 5.1 核心发现

1. **"制度性审核"是真正创新**：门下省的封驳机制解决的是"AI 输出质量不可控"的问题。CrewAI/AutoGen 没有这个机制，是因为它们假设 AI 输出总是好的——这是个危险假设。

2. **历史智慧的现代应用**：作者用唐朝的三省六部制度做隐喻，不是为了酷，而是因为这个制度本身就是一套经过 1400 年验证的**分权制衡机制**。放到 AI Agent 协作里，逻辑完全成立。

3. **看板是第一公民**：军机处看板不是附加功能，而是和 Agent 协作平级的核心功能。这种"可观测性"在现有 Multi-Agent 框架里普遍缺失。

4. **零外部依赖后端是个性化选择**：stdlib only 意味着不依赖 LangChain/AutoGen 等框架，可以更灵活地定制。但也意味着更多定制开发工作。

5. **"奏折"概念是巧妙的 UX**：把完成任务叫"回奏"，把归档叫"奏折阁"——用中国古代的文书流程做隐喻，让用户更容易理解任务流转。

### 5.2 局限性

- **依赖 OpenClaw**：不是独立框架，需要 OpenClaw 才能运行
- **中国文化背景**：对中国古代官制不熟悉的用户可能觉得概念难懂
- **六部职能划分**：实际项目中，任务不一定能明确划分到六部
- **学习成本**：比简单框架更复杂，需要理解三省六部的运作逻辑

### 5.3 架构评分

| 维度 | 评分 | 说明 |
|------|------|------|
| **审核机制** | ★★★★★ | 门下省封驳是真正创新 |
| **可观测性** | ★★★★★ | 10 面板军机处看板 |
| **可干预性** | ★★★★★ | 叫停/取消/恢复 |
| **架构严谨性** | ★★★★☆ | 三省六部分权制衡，逻辑自洽 |
| **部署体验** | ★★★★★ | Docker/一键安装 |
| **创新性** | ★★★★★ | 历史智慧 + 现代 AI |

**综合评价**：edict 三省六部是 AI Agent 协作框架中**最有思想深度**的项目。它不是又一个"CrewAI-like"的框架，而是真正思考了"AI Agent 协作需要什么制度保障"。门下省的强制审核机制解决的是真实问题——AI 输出不可控。军机处看板解决了可观测性。分权制衡设计让整个系统既高效又安全。对于需要严肃对待 AI 输出质量的团队来说，这是一个值得关注的项目。
