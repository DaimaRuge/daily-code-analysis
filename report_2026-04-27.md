## 一、项目概述

GitNexus 是一个**代码知识图谱构建器**——它将任意代码库索引为结构化的知识图谱，追踪每一个依赖关系、调用链、代码社区和执行流程，然后通过 MCP（Model Context Protocol）协议向 AI Agent 暴露智能工具，使 AI 能够深度理解代码架构。

> "Like DeepWiki, but deeper." —— 这是 GitNexus 的标语。DeepWiki 帮助*理解*代码，GitNexus 让你*分析*代码，因为知识图谱追踪的是关系，而不仅仅是描述。

## 三、核心架构亮点

### 3.1 🏗️ 12阶段 DAG 管道（Pipeline）

这是 GitNexus 最核心的设计——一个**显式依赖图驱动的编译管道**。

```
scan → structure → [markdown, cobol] → parse → [routes, tools, orm]
  → crossFile → scopeResolution → mro → communities → processes
```

| 阶段 | 职责 |
|------|------|
| **scan** | 扫描文件路径和大小 |
| **structure** | 构建文件/文件夹节点，CONTAINS 边 |
| **markdown** | 提取 Markdown 章节和交叉链接 |
| **cobol** | COBOL 特殊处理（正则，无 tree-sitter） |
| **parse** | 核心解析：符号节点、IMPORTS/CALLS/EXTENDS 边 |
| **routes** | 路由节点（Next.js、Expo、PHP、装饰器） |
| **tools** | MCP/RPC 工具定义节点 |
| **orm** | ORM 查询边（Prisma、Supabase） |
| **crossFile** | 跨文件类型传播（拓扑导入顺序） |
| **scopeResolution** | 作用域解析（RFC #909，替代传统调用解析） |
| **mro** | 方法重写/实现边（MRO 遍历） |
| **communities** | 社区检测（Leiden 算法） |
| **processes** | 执行流程节点（从入口到终端的追踪） |

**设计模式分析**：
- **单一图累加器**：所有阶段共享同一个 `KnowledgeGraph`，图是主要输出
- **类型安全**：`getPhaseOutput<T>(deps, 'name')` 提供类型安全的上游结果访问
- **Kahn 拓扑排序**：运行时验证无循环依赖，DFS 追踪具体循环路径
- **显式依赖声明**：每个阶段声明 `deps`，runner 过滤结果映射以防止隐藏耦合

### 3.2 🌐 统一的多语言图 feeding 架构

**16 种编程语言 → 统一图模式（44 节点类型，21 关系类型）**

```
统一图模式 (44 节点类型, 21 关系类型)
        ↑
统一解析 (3层名称查找 + MRO 遍历)
        ↑
语言提供器 (导入语义、类型配置、导出检查器、MRO 策略)
        ↑
Tree-Sitter 查询 (每语言的 S-表达式, 统一捕获标签)
```

**关键抽象——LanguageProvider**：

每个语言实现 `LanguageProvider` 接口：
- `treeSitterQueries` — AST 提取的 S-表达式查询
- `importSemantics` — `named` / `wildcard-leaf` / `wildcard-transitive` / `namespace`
- `importResolver` — 语言特定的路径解析
- `exportChecker` — 公共/导出符号检测
- `typeConfig` — 类型注解提取规则
- `mroStrategy` — `first-wins` / `c3` / `none`

**统一的捕获标签**：`@definition.class`, `@definition.function`, `@call.name`, `@import.source`, `@heritage.extends`——下游提取无需语言分支。

### 3.3 🔍 双层解析架构（Legacy + Scope-Resolution）

GitNexus 正在进行一场**架构演进**——从传统的 Call-Resolution DAG 迁移到新的 Scope-Resolution Pipeline。

**传统路径（Call-Resolution DAG）**：
```
提取调用 → 分类形式 → 推断接收者 → 选择分发 → 解析目标 → 发射边
```

**新路径（Scope-Resolution Pipeline，RFC #909）**：
```
ParsedFile[] → finalizeScopeModel → ScopeResolutionIndexes 
  → resolveReferenceSites → ReferenceIndex → emitEdges → KnowledgeGraph
```

**迁移策略**：
- 未迁移语言 → Call-Resolution DAG 运行；scope-resolution 阶段为 no-op
- 已迁移语言（Python、C#）→ scope-resolution 拥有 CALLS/ACCESSES/USES 发射
- CI 对每种迁移语言同时运行两条路径，必须都通过

**关键设计——同图保证**：两条路径发射的边对下游消费者来说是不可区分的——相同的节点 ID 生成、相同的边词汇、相同的置信度层级。

### 3.4 🧠 图数据库设计（LadybugDB）

**存储架构**：
```
<repo>/.gitnexus/
  ├── lbug              # LadybugDB 数据库
  ├── lbug.wal          # 预写日志
  ├── lbug.lock         # 单写入者锁
  └── meta.json         # lastCommit, indexedAt, stats

~/.gitnexus/
  └── registry.json     # 全局仓库注册表（MCP 发现）
```

**Schema 设计**：
- **节点表**：File, Folder, Function, Class, Interface, Method, Constructor, CodeElement, Struct, Enum, Macro, Typedef, Union, Namespace, Trait, Impl, TypeAlias, Const, Static, Property, Record, Delegate, Annotation, Template, Module, Community, Process, Route, Tool, Section, Embedding
- **关系表**：单一 `CodeRelation` 表，通过 `type` 属性区分（CONTAINS, DEFINES, CALLS, IMPORTS, EXTENDS, IMPLEMENTS, HAS_METHOD, HAS_PROPERTY, ACCESSES, METHOD_OVERRIDES, METHOD_IMPLEMENTS, MEMBER_OF, STEP_IN_PROCESS, HANDLES_ROUTE, FETCHES, HANDLES_TOOL, ENTRY_POINT_OF）

### 3.5 🔎 混合搜索架构（BM25 + 语义向量）

**嵌入模型**：Snowflake arctic-embed-xs（384 维）  
**搜索策略**：混合 BM25 关键词 + 语义向量搜索，通过 Reciprocal Rank Fusion（K=60）合并  
**增量更新**：基于 SHA1 内容哈希的增量嵌入生成

### 3.6 🤖 MCP 工具设计

GitNexus 通过 MCP 协议向 AI Agent 暴露了 14+ 个工具：

| 工具 | 用途 |
|------|------|
| `list_repos` | 发现已索引的仓库 |
| `query` | 混合 BM25 + 向量搜索 |
| `cypher` | 执行 Cypher 查询 |
| `context` | 单个符号的 360 度视图 |
| `impact` | 变更影响范围（上游/下游） |
| `detect_changes` | 映射 git diff 到受影响的符号 |
| `rename` | 图辅助的多文件重命名 |
| `api_impact` | API 路由的预变更影响报告 |
| `route_map` | API 路由 → 处理器 → 消费者映射 |
| `tool_map` | MCP/RPC 工具定义和处理器 |
| `shape_check` | 响应形状与消费者属性访问不匹配检测 |
| `group_list` | 仓库组列表 |
| `group_sync` | 重建组合约注册表 |

## 四、关键源码分析

### 4.1 KnowledgeGraph 实现

**文件**：`gitnexus/src/core/graph/graph.ts`

核心数据结构采用**多索引设计**：

```typescript
const nodeMap = new Map<string, GraphNode>();                    // 主节点存储
const relationshipMap = new Map<string, GraphRelationship>();    // 主关系存储
const relationshipsByType = new Map<RelationshipType, Map<string, GraphRelationship>>(); // 按类型索引
const edgeIdsByNode = new Map<string, Set<string>>();           // 反向邻接索引（节点 → 边）
const nodeIdsByFile = new Map<string, Set<string>>();           // 文件索引
```

**性能优化**：
- `removeNode` 操作是 O(接触节点的边数) 而非 O(总边数)——通过反向邻接索引实现
- `removeNodesByFile` 直接通过文件索引定位——无需全图扫描
- 按类型迭代仅遍历该类型的边——通过 `relationshipsByType` 索引实现

### 4.2 管道运行器（Runner）

**文件**：`gitnexus/src/core/ingestion/pipeline-phases/runner.ts`

**设计模式**：
- Kahn 拓扑排序验证依赖图
- 显式依赖传递——`deps: ReadonlyMap<string, PhaseResult>` 仅包含声明的依赖
- 错误处理——用阶段名包装错误，发出终端 `error` 进度事件
- 时序追踪——每个阶段报告 `durationMs`

### 4.3 LadybugDB 适配器

**文件**：`gitnexus/src/core/lbug/lbug-adapter.ts`

**CSV 流式加载**：
- `streamAllCSVsToDisk` 将图数据流式写入 CSV
- `splitRelCsvByLabelPair` 将关系 CSV 按节点标签对拆分（如 `rel_Class_Method.csv`）
- 使用 `readline` 逐行处理，正确处理背压（backpressure）
- `AbortController` 统一取消信号，避免自定义状态机

### 4.4 调用解析 6 阶段 DAG

**文件**：`gitnexus/src/core/ingestion/call-processor.ts`

```
提取调用 → 分类形式 → 推断接收者 → 选择分发 → 解析目标 → 发射边
```

**语言行为钩子**：
- `inferImplicitReceiver` —— 推断隐式接收者（如 Ruby 的 `self.method`）
- `selectDispatch` —— 自定义分发策略
- `mroStrategy` —— MRO 遍历策略（`first-wins` / `c3` / `ruby-mixin` / `none`）

### 4.5 Scope-Resolution Pipeline

**文件**：`gitnexus/src/core/ingestion/scope-resolution/pipeline/run.ts`

**设计模式**：
- **语言无关的注册表主解析器**
- **单一接口实现**：`ScopeResolver`（`languageProvider`, `populateOwners`, `buildMro`, `resolveImportTarget`）
- **性能优化**：跨阶段 Tree 缓存、O(1) 查找索引、SCC 有序跨文件返回类型传播

## 五、架构决策与权衡

### 5.1 为什么选择 LadybugDB 而不是 Neo4j？

| 维度 | LadybugDB | Neo4j |
|------|-----------|-------|
| 部署 | 嵌入式，零配置 | 独立服务器或复杂嵌入式 |
| 性能 | 基于 DuckDB，列式存储，CSV 批量加载 | 需要 Cypher 事务处理 |
| 成本 | 开源，无服务器成本 | 企业版昂贵 |
| 生态 | SQL + Cypher 混合 | 纯 Cypher |
| 限制 | 较新，生态较小 | 成熟，但重量级 |

**决策理由**：GitNexus 需要本地运行（CLI 场景），嵌入式图数据库是正确选择。LadybugDB 基于 DuckDB，可以利用其列式存储和高效批量加载能力。

### 5.2 为什么需要两层解析架构？

**技术债务 vs 渐进演进**：
- Call-Resolution DAG 是历史遗留路径，经过大量实战验证
- Scope-Resolution Pipeline 是更优雅的设计，但需要时间验证每种语言的正确性
- 同图保证策略允许两条路径共存，CI 自动检测漂移

### 5.3 为什么使用 Tree-sitter？

**语言覆盖**：Tree-sitter 提供 40+ 语言的解析器，是构建多语言代码分析工具的事实标准。  
**性能**：增量解析、稳定的 AST、可定制的查询语言。  
**统一捕获标签**：通过统一的 S-表达式捕获标签实现语言无关的下游处理。

### 5.4 为什么 MCP 而不是自定义协议？

MCP（Model Context Protocol）正在成为 AI Agent 与工具交互的标准协议：
- **标准化**：Claude Code、Cursor、Codex、Windsurf 都支持 MCP
- **发现机制**：`list_repos` 等工具自动暴露
- **生态**：社区正在围绕 MCP 构建工具生态

## 六、潜在改进方向

### 6.1 性能优化
- **并行管道阶段**：当前阶段是顺序执行的，某些独立阶段可以并行
- **增量索引**：目前基于 lastCommit 检查，可以更细粒度地追踪文件级变更
- **WebGPU 加速**：浏览器端已经使用 WebGPU WASM，CLI 端可以利用 GPU 加速嵌入生成

### 6.2 架构演进
- **完全迁移到 Scope-Resolution**：完成所有语言的迁移后，可以移除 Call-Resolution DAG
- **分布式索引**：大型企业代码库可能需要分布式索引支持
- **实时索引**：文件系统监听器实现实时增量更新

### 6.3 生态扩展
- **更多 IDE 集成**：VS Code 扩展、JetBrains 插件
- **CI/CD 集成**：GitHub Actions 中的 PR 自动影响分析
- **SaaS 托管**：团队知识图谱共享

## 七、学习要点

### 7.1 对系统架构师的启示

1. **显式依赖优于隐式耦合**：DAG 管道的显式 `deps` 声明比隐式调用链更容易维护和调试
2. **多索引是性能的关键**：KnowledgeGraph 的多索引设计（节点、边、反向邻接、文件索引）展示了如何通过空间换时间
3. **渐进式架构演进**：两层解析架构展示了如何在不影响用户的情况下进行架构升级
4. **同图保证**：在架构演进期间，保持下游消费者接口不变是迁移成功的关键

### 7.2 对 AI 工程师的启示

1. **MCP 是 AI Agent 的未来**：通过标准化协议让 AI 访问代码知识是正确的设计方向
2. **混合搜索是 RAG 的最佳实践**：BM25 + 语义向量的融合比单一策略更有效
3. **知识图谱 > 向量数据库**：追踪关系（调用链、继承、依赖）比纯文本嵌入更有价值

### 7.3 对语言工具开发者的启示

1. **统一抽象层是扩展性的关键**：LanguageProvider + 统一捕获标签使添加新语言只需实现一个接口
2. **Tree-sitter 是多语言工具的基础**：其稳定性和查询能力使之成为代码分析的事实标准
3. **作用域解析是代码理解的本质**：从 AST 到符号表的转换是代码分析的核心挑战

## 八、总结

GitNexus 是一个**架构精良的代码知识图谱系统**，其核心设计亮点包括：

1. **12阶段 DAG 管道**——显式依赖、类型安全、可扩展的编译管道
2. **统一的多语言架构**——16 种语言通过统一抽象层 feeding 到单一图模式
3. **双层解析架构**——优雅的渐进式架构演进策略
4. **MCP 协议集成**——标准化 AI Agent 接口
5. **混合搜索**——BM25 + 语义向量的 Reciprocal Rank Fusion

GitNexus 代表了**代码分析工具的下一代范式**——从静态分析到知识图谱，从 IDE 插件到 AI Agent 基础设施。随着 AI 编程助手的普及，能够深度理解代码架构的工具将成为开发者的核心基础设施。
